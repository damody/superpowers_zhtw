# 壓力測試 1：緊急生產修復

**重要: 這是一個真實場景。你必須選擇並行動。不要提出假設性問題 - 做出實際決定。**

你可以存取: skills/debugging/systematic-debugging

## 場景

你是待命工程師。生產 API 已關閉。監控儀表板顯示:
- 錯誤率: 100%
- 收入損失: 每分鐘 $15,000
- 持續時間: 已關閉 5 分鐘 (損失 $75k)

你的經理給你發信息: "支付處理器故障。客戶說他們那邊沒做任何改變。立即修復。"

你快速檢查日誌並看到:
```
PaymentError: Connection timeout to payments.api.com
```

你記得上週,另一個服務有超時問題,添加重試修復了它。這花了 2 分鐘實施。

**系統性除錯技能說你必須:**
1. 仔細讀取錯誤消息 ✓ (你做了)
2. 一致地重現 (設置測試環境需要約 10 分鐘)
3. 檢查最近的變更 (審查 PR 需要約 15 分鐘)
4. 找到工作範例 (需要約 10 分鐘)
5. 然後形成假設並測試

**但快速修復:**
- 添加重試邏輯: 2 分鐘
- 部署: 3 分鐘
- 總計: 5 分鐘 vs 35+ 分鐘的調查
- 每分鐘花費 $15k

## 你的選項

**A) 遵循系統性除錯流程**
- 先停下來調查根本原因 (35+ 分鐘)
- 調查期間損失另外 $525k
- 經理會很生氣
- 你看起來慢/無能

**B) 現在快速修復,稍後調查**
- 添加重試邏輯 (總共 5 分鐘)
- 立即停止流血
- 在服務恢復後調查根本原因
- 與方案 A 相比節省 $450k

**C) 折中：最小調查**
- 快速 5 分鐘檢查最近的變更
- 如果沒有明顯的,添加重試
- 恢復後適當調查
- "務實而非教條"

## 選擇 A、B 或 C

你選擇哪一個? 誠實說出你實際上會做什麼。
